Как устроена система F&C

Система управленческого учета F&C предназначена для автоматического формирования управленческой отчетности на основании единых исходных данных.

Основной принцип системы заключается в том, что каждая хозяйственная операция вводится только один раз, после чего автоматически используется при формировании всех отчетов.

Благодаря этому исключается повторный ввод информации, снижается вероятность ошибок и обеспечивается единый источник данных для всей управленческой отчетности.

Блок 1. Архитектура системы ▼

Система состоит из двух взаимосвязанных файлов.

📄 Файл «УУ_данные»

Это основной рабочий файл финансового менеджера. Именно в нем выполняется вся ежедневная работа:

  • загрузка банковских выписок;
  • заполнение операций;
  • определение контрагентов;
  • выбор субстатей;
  • внесение начислений;
  • проверка корректности данных.

Все изменения в системе выполняются только в этом файле.

📊 Файл «УУ_финотчеты»

Этот файл предназначен только для формирования управленческой отчетности. На основании данных из файла «УУ_данные» автоматически формируются:

  • Отчет о движении денежных средств (ДДС);
  • Отчет о прибылях и убытках (ОПиУ).

Изменять данные в этом файле запрещено. Если обнаружена ошибка в отчетах, необходимо найти и исправить ее в файле «УУ_данные».

Как работает система

Работа системы построена по следующему принципу:

Первичные данные │ ▼ Файл «УУ_данные» │ ▼ База данных (БД) │ ▼ Файл «УУ_финотчеты» │ ├── ДДС └── ОПиУ

Финансовый менеджер работает только с первичными данными. Все последующие расчеты, группировки и отчеты формируются автоматически.

Главный принцип работы

При возникновении любой ошибки необходимо помнить простое правило: исправления всегда вносятся в источник данных, а не в отчеты. Если показатель в ДДС или ОПиУ рассчитан неверно, необходимо:

  1. Определить, из какого источника поступают данные.
  2. Исправить информацию в соответствующем рабочем листе файла «УУ_данные».
  3. Проверить, что изменения автоматически отразились в отчетах.
Важно: Всегда исправляйте причину ошибки, а не её последствия. Если ошибка в отчёте — правьте данные в источнике.
Блок 2. Рабочие файлы системы ▼

Для ведения управленческого учета используются два взаимосвязанных файла, каждый из которых выполняет свою функцию. Очень важно понимать назначение каждого файла, так как ошибки чаще всего возникают из-за попытки внести изменения не в том месте.

Файл «УУ_данные»

Это основной рабочий файл, с которым финансовый менеджер работает ежедневно. Все хозяйственные операции первоначально заносятся именно сюда. В этом файле находятся все рабочие листы, необходимые для формирования управленческой отчетности. Именно здесь выполняется:

  • загрузка банковских выписок;
  • определение контрагентов;
  • выбор субстатей;
  • проверка правильности классификации операций;
  • внесение начислений;
  • корректировка данных при обнаружении ошибок.

Все изменения в системе выполняются только в файле «УУ_данные».

Файл «УУ_финотчеты»

Этот файл предназначен только для анализа и проверки результатов. На основании данных, содержащихся в файле «УУ_данные», автоматически формируются:

  • Отчет о движении денежных средств (ДДС);
  • Отчет о прибылях и убытках (ОПиУ).

В каждом отчете предусмотрено два варианта представления данных:

  • Сводная таблица — используется для анализа структуры данных и поиска ошибок.
  • Отчет по формулам — используется как основной управленческий отчет для руководства.

Итоговые показатели обоих вариантов должны совпадать.

Основной принцип работы

При обнаружении любой ошибки необходимо определить, в каком источнике данных она возникла. Исправлять необходимо причину ошибки, а не ее последствия. Например:

  • если неверно указана субстатья — исправление выполняется в банковском листе;
  • если неправильно отражено начисление — исправление выполняется в листе «Источник начисления» или в банковском листе;
  • если требуется изменить классификацию операции — изменения вносятся в «Справочник статей».

После исправления исходных данных система автоматически пересчитает все связанные отчеты.

Что входит в файл «УУ_данные»

Рабочий файл состоит из нескольких групп листов.

1. Банковские листы

Используются для переноса операций из банковских выписок. В вашей системе это:

  • BAI;
  • BCA;
  • BI;
  • CECV;
  • BAI VISA;
  • BAI POUP-2;
  • BAI POUP-5;
  • CECV POUP.

Несмотря на разные названия, все банковские листы заполняются по одинаковым правилам.

2. Справочники

Используются для хранения постоянной информации, необходимой для автоматической работы системы. Например:

  • контрагенты;
  • статьи;
  • субстатьи;
  • настройки классификации операций.

3. База данных (БД)

База данных объединяет информацию из всех источников в единый массив. Финансовый менеджер не заполняет этот лист вручную. Он используется только для проверки, что все операции корректно попали в систему.

4. Источник начисления

Используется для отражения операций, которые невозможно получить из банковских выписок или которые должны отражаться в ОПиУ по методу начисления. К таким операциям относятся:

  • выручка;
  • себестоимость;
  • начисленная заработная плата (если не отображаться в выписках, перепроверить и скорректировать при появлении в выписке, чтобы не было задвоения);
  • начисления налогов работодателя;
  • другие корректировки.
Важно: Файл «УУ_данные» — это место, где создаются данные. Файл «УУ_финотчеты» — это место, где анализируются результаты. Если придерживаться этого принципа, большинство ошибок в работе системы удастся избежать.
Блок 3. Работа с банковскими листами ▼

Работа с банковскими выписками является первым этапом формирования управленческой отчетности. Именно на этом этапе закладывается качество всех последующих отчетов. Ошибки, допущенные при переносе банковских операций, приводят к некорректному формированию ДДС, ОПиУ, а также других управленческих отчетов. Поэтому основной принцип работы заключается в следующем: сначала необходимо правильно определить экономический смысл операции, и только после этого выбирать статью учета. Не следует ориентироваться только на текст назначения платежа в банковской выписке. Одинаковые банковские описания могут относиться к совершенно разным хозяйственным операциям.

Какие банковские листы используются

В системе каждый банковский счет имеет отдельный рабочий лист. Например:

  • BAI;
  • BCA;
  • BI;
  • CECV;
  • BAI VISA;
  • BAI POUP-2;
  • BAI POUP-5;
  • CECV POUP.

Несмотря на разные названия банковских счетов, порядок работы со всеми листами полностью одинаковый. После получения банковской выписки операции переносятся в соответствующий банковский лист без изменения сумм и дат.

Общий порядок работы

Работа с банковскими выписками выполняется в следующей последовательности.

Шаг 1. Получить банковскую выписку

Перед началом работы необходимо убедиться, что выписка получена полностью за выбранный период.

Шаг 2. Перенести операции

Все операции из банковской выписки переносятся в соответствующий банковский лист. При переносе необходимо сохранить исходные значения:

  • дату операции;
  • описание операции;
  • сумму поступления или списания.

Изменять эти данные не рекомендуется, но если суммы разделяются на несколько платежей — внимательно все пересчитать, чтобы не было искажения данных.

Шаг 3. Определить контрагента

После переноса операции необходимо определить контрагента. Если контрагент уже существует в системе, необходимо выбрать его из справочника. Если контрагент отсутствует, его следует добавить в лист «Справочники» в соответствии с внутренними правилами компании. Контрагент должен быть указан максимально точно, так как эта информация используется при анализе задолженности и подготовке управленческих отчетов.

Самое важное поле — «Субстатья»

После определения контрагента необходимо выбрать Субстатью. Это самое важное действие. Именно субстатья определяет дальнейшую обработку операции системой. На основании выбранной субстатьи автоматически определяются:

  • статья ОДДС;
  • раздел ОДДС;
  • статья ОПиУ;
  • раздел ОПиУ;
  • необходимость участия операции в управленческой отчетности;
  • правила определения даты начисления;
  • дальнейшая классификация операции.

Поэтому выбор субстатьи должен производиться осознанно, исходя из экономического смысла операции. Не рекомендуется выбирать субстатью только по тексту назначения платежа.

Почему нельзя ориентироваться только на банковскую выписку

Банк отражает только движение денежных средств. Однако одна и та же операция может иметь различное экономическое содержание. Например, поступление денежных средств может означать:

  • оплату покупателем ранее отгруженного товара;
  • получение аванса;
  • возврат займа;
  • получение кредита;
  • внутренний перевод между собственными счетами.

Хотя в банковской выписке все эти операции выглядят как поступление денежных средств, в управленческом учете они отражаются по-разному. Именно поэтому необходимо понимать экономический смысл каждой операции.

После заполнения банковского листа

После обработки всех операций необходимо выполнить контроль. Следует проверить:

  • все ли операции перенесены;
  • заполнены ли контрагенты;
  • заполнены ли субстатьи;
  • отсутствуют ли пустые обязательные поля, например: Счет, Валюта и т.д.;
  • нет ли операций с некорректными суммами или датами.

Только после этого можно переходить к следующему этапу работы.

Типичные ошибки

Ошибка №1. Не заполнена субстатья

Последствие: операция не сможет быть корректно классифицирована системой. В результате могут не заполниться статьи ОДДС, статьи ОПиУ и другие связанные поля.

Ошибка №2. Выбрана похожая, но неверная субстатья

Последствие: операция попадет в неправильный раздел отчетности. Даже если сумма верная, финансовый результат будет искажен.

Ошибка №3. Изменена сумма операции

Последствие: данные перестанут соответствовать банковской выписке.

Ошибка №4. Попытка исправить отчет вместо источника данных

Если после формирования отчетности обнаружена ошибка, нельзя исправлять итоговые показатели вручную. Необходимо найти ошибочную операцию в банковском листе, исправить ее и убедиться, что изменения автоматически отразились в БД и отчетах.

Итог раздела
После завершения работы с банковскими листами следует быть уверенными:
  • все операции из выписки перенесены;
  • каждому платежу определен контрагент;
  • каждой операции присвоена правильная субстатья;
  • данные полностью соответствуют банковской выписке;
  • система готова к дальнейшему формированию Базы данных и управленческой отчетности.
Блок 4. Работа со «Справочником статей» ▼
Назначение листа

Лист «Справочник статей» является одним из ключевых элементов системы управленческого учета F&C. Именно в этом листе определяется логика отражения каждой хозяйственной операции в управленческой отчетности. Если банковские листы отвечают на вопрос «Что произошло?», то Справочник статей отвечает на вопрос «Как эта операция должна отражаться в системе?» Другими словами, этот лист является связующим звеном между банковскими операциями и управленческой отчетностью.

Для чего нужен Справочник статей

Каждая банковская операция содержит только первичную информацию:

  • дату;
  • сумму;
  • назначение платежа;
  • контрагента.

Однако этого недостаточно для формирования управленческой отчетности. Система должна понимать:

  • является ли операция поступлением или выбытием денежных средств;
  • должна ли она попасть в ДДС;
  • должна ли она попасть в ОПиУ;
  • к какому разделу относится;
  • в каком периоде должна быть отражена;
  • участвует ли операция в формировании финансового результата.

Вся эта информация определяется автоматически на основании Субстатьи.

Как работает Справочник статей

После выбора субстатьи система обращается к листу «Справочник статей» и автоматически определяет все остальные параметры операции. Например, если выбрана субстатья «Оплата поставщику за товар», система автоматически понимает:

  • Вид операции — Выбытие;
  • Раздел ОДДС — Операционная деятельность;
  • Статья ОДДС — Оплата поставщикам;
  • Статья ОПиУ — Не учитывать (если это только движение денежных средств);
  • Дальнейшую логику обработки операции.

Финансовому менеджеру не требуется заполнять эти поля вручную.

Какие поля настраиваются в Справочнике статей

Для каждой субстатьи задаются параметры, определяющие дальнейшее отражение операции. Как правило, для каждой субстатьи указываются:

  • Вид операции;
  • Раздел ОДДС;
  • Статья ОДДС;
  • Раздел ОПиУ;
  • Статья ОПиУ;
  • Признак участия в отчетах;
  • Дополнительные настройки, используемые системой.

Именно поэтому изменение справочника влияет сразу на всю систему.

Когда необходимо изменять Справочник статей

В повседневной работе финансовый менеджер не должен регулярно изменять этот лист. Изменения вносятся только в следующих случаях:

  • появилась новая хозяйственная операция, для которой отсутствует субстатья;
  • изменились правила отражения операций по согласованию с финансовым директором;
  • требуется добавить новую статью управленческого учета.

Если существующая субстатья подходит по своему экономическому смыслу, создавать новую не следует.

Как добавить новую субстатью

Перед созданием новой записи необходимо убедиться, что аналогичной субстатьи еще нет. Если подходящая субстатья отсутствует:

  1. определить экономический смысл новой операции;
  2. выбрать раздел ОДДС;
  3. определить статью ОДДС;
  4. определить необходимость участия операции в ОПиУ;
  5. указать статью ОПиУ (или выбрать «Не учитывать», если операция не влияет на финансовый результат).

Только после этого новая субстатья может использоваться в банковских листах.

Особое внимание — статье «Не учитывать»

В системе F&C существует специальное значение «Не учитывать». Оно используется в тех случаях, когда операция должна присутствовать в одном отчете, но не должна участвовать в другом. Например, получение банковского кредита: денежные средства поступили на расчетный счет. Следовательно, операция должна попасть в ДДС. Но кредит не является доходом компании. Поэтому в ОПиУ эта операция отражаться не должна. В этом случае для статьи ОПиУ выбирается значение «Не учитывать». Аналогичная логика применяется к:

  • внутренним переводам между счетами;
  • возврату займа;
  • другим операциям, не влияющим на финансовый результат.
Почему нельзя создавать похожие субстатьи

Одна из самых распространенных ошибок — создание нескольких субстатей с одинаковым экономическим смыслом. Например:

  • Оплата поставщику;
  • Оплата поставщику товара;
  • Оплата за товар поставщику;
  • Оплата товара.

Если все эти субстатьи описывают одну и ту же операцию, использовать их одновременно не следует. Это затрудняет анализ данных, усложняет поиск ошибок и приводит к дублированию настроек. Для каждой типовой хозяйственной операции рекомендуется использовать одну утвержденную субстатью.

Контроль после внесения изменений

После добавления новой субстатьи необходимо проверить:

  • корректно ли определяются статьи ОДДС;
  • правильно ли определяется статья ОПиУ;
  • участвует ли операция в нужных отчетах;
  • не возникает ли ошибок в Базе данных;
  • корректно ли отражается операция в ДДС и ОПиУ.

Только после такой проверки новая субстатья может использоваться в ежедневной работе.

Запомните: Справочник статей — это не справочник названий. Это правила работы всей системы. Любое изменение в этом листе автоматически влияет на классификацию операций, формирование ДДС, ОПиУ и другие управленческие отчеты. Поэтому изменения в справочник должны вноситься осознанно и только после понимания того, как они повлияют на всю систему.
Блок 5. Работа с листом «БД» ▼
Назначение листа

Лист «БД» (База данных) является центральным элементом системы управленческого учета F&C. Именно в этом листе объединяются данные из всех рабочих источников:

  • банковских листов;
  • листа «Источник начисления»;
  • других источников данных, используемых системой.

База данных является единым массивом информации, на основании которого формируются все управленческие отчеты. Поэтому лист «БД» можно сравнить с сердцем системы: если информация не попала в Базу данных, значит, она не попадет ни в один отчет.

Для чего нужна База данных

Во время ежедневной работы информация вводится в разные листы. Например:

  • банковские операции — в банковские листы;
  • начисления — в лист «Источник начисления».

Для формирования отчетности все эти данные должны быть объединены в единую таблицу. Именно эту функцию выполняет лист «БД». Он автоматически собирает все операции в едином формате и передает их в файл «УУ_финотчеты». Данные не переносятся в БД вручную. Все записи появляются автоматически.

Главная задача финансового менеджера

Работа с листом «БД» заключается не в заполнении, а в контроле качества данных. После внесения новых операций необходимо убедиться, что:

  • операция появилась в Базе данных;
  • все обязательные поля заполнены;
  • отсутствуют ошибки формул;
  • корректно определились статьи и разделы учета.

Таким образом, лист «БД» используется как основной инструмент проверки правильности работы системы.

Когда необходимо открывать лист «БД»

Рекомендуется проверять Базу данных после каждого значимого этапа работы. Например:

  • после загрузки банковских выписок;
  • после внесения новых начислений;
  • после изменения справочников;
  • перед формированием отчетности;
  • перед закрытием месяца.

Такая последовательность позволяет обнаружить ошибки до формирования отчетов.

Что необходимо проверить

При открытии листа «БД» необходимо последовательно проверить следующие вопросы.

1. Все ли операции появились в Базе данных?

Если часть операций отсутствует, необходимо проверить соответствующий источник данных. Например:

  • банковский лист;
  • лист «Источник начисления».

2. Нет ли пустых обязательных полей?

Особое внимание необходимо обратить на:

  • Субстатью;
  • Контрагента;
  • Статью ОПиУ;
  • Раздел ОДДС;
  • Дату начисления (если она должна быть заполнена);
  • Счет.

Пустые обязательные поля свидетельствуют о том, что операция была заполнена не полностью или в системе отсутствуют необходимые настройки.

3. Правильно ли определились статьи?

После выбора субстатьи система автоматически определяет:

  • Вид операции;
  • Раздел ОДДС;
  • Статью ОДДС;
  • Раздел ОПиУ;
  • Статью ОПиУ.

Если хотя бы одно из этих полей заполнено неверно, необходимо проверить настройки Справочника статей.

4. Нет ли дублирования операций?

Каждая хозяйственная операция должна присутствовать в Базе данных только один раз. Если одна и та же операция отображается дважды, необходимо определить причину:

  • повторная загрузка банковской выписки;
  • дублирование записи в источнике данных;
  • ошибка при объединении данных.
Что категорически запрещено

Лист «БД» является автоматически формируемым листом. Поэтому запрещается:

  • изменять данные вручную;
  • удалять строки;
  • добавлять строки;
  • изменять формулы;
  • исправлять статьи непосредственно в Базе данных.

Любые изменения должны выполняться только в соответствующем источнике данных.

Что делать, если обнаружена ошибка

Если при проверке Базы данных обнаружена ошибка, необходимо определить источник ее возникновения.

Если ошибка связана с... Проверить
отсутствием операции соответствующий банковский лист или лист «Источник начисления»
неправильной статьей лист «Справочник статей»
отсутствием контрагента лист «Справочники»
отсутствием даты начисления исходную операцию и правила определения даты начисления

После исправления исходных данных необходимо убедиться, что изменения автоматически отразились в листе «БД».

Типичные ошибки

Ошибка №1. Попытка исправить данные прямо в Базе данных

Это одна из самых распространенных ошибок. Такое исправление не устраняет причину проблемы и может нарушить работу системы.

Ошибка №2. Не проверена База данных после загрузки выписки

Если ошибка не обнаружена на этом этапе, она автоматически попадет в ДДС, ОПиУ и другие отчеты.

Ошибка №3. Игнорирование пустых обязательных полей

Если система не смогла автоматически определить статью или раздел, значит, одна из предыдущих настроек выполнена некорректно. Такие ошибки необходимо устранять сразу.

Запомните: База данных не является рабочим листом для ввода информации. Это контрольный лист, который позволяет убедиться, что система корректно обработала все введенные операции. Если операция отсутствует в Базе данных, дальнейшая проверка отчетов не имеет смысла — сначала необходимо найти и устранить причину в источнике данных.
Итог раздела
После проверки листа «БД» вы должны быть уверены, что:
  • все операции успешно перенесены;
  • обязательные поля заполнены;
  • классификация операций выполнена корректно;
  • отсутствуют дубли и ошибки;
  • система готова к формированию управленческой отчетности.
Блок 6. Работа с листом «Источник начисления» ▼
Назначение листа

Лист «Источник начисления» используется для отражения операций, которые невозможно корректно сформировать только на основании банковских выписок. Одной из главных особенностей управленческого учета является то, что движение денежных средств и финансовый результат компании не всегда происходят одновременно. Поэтому не все показатели ОПиУ могут быть сформированы по данным банковских операций. Для таких операций используется лист «Источник начисления». Его задача — обеспечить правильное отражение доходов и расходов в том периоде, к которому они относятся, независимо от даты оплаты.

Почему недостаточно банковской выписки

Если деньги уже поступили или были оплачены через банк, зачем нужен еще один лист? Ответ заключается в том, что банк отражает только движение денежных средств. Но управленческий учет отвечает на другой вопрос: когда компания действительно заработала доход или понесла расход? Это не всегда совпадает с датой оплаты.

Пример №1. Выручка

Покупатель оплатил товар 25 января. Однако товар был отгружен только 5 февраля. В банковской выписке поступление денежных средств появится в январе. Но доход компания получит только после отгрузки товара. Поэтому в ОПиУ выручка должна быть отражена в феврале. Именно поэтому данные по выручке берутся не из банковской выписки, а из системы учета продаж (Vendus).

Пример №2. Себестоимость

Оплата поставщику могла быть произведена несколько месяцев назад. Однако товар был реализован только в текущем месяце. Расход в виде себестоимости возникает именно в момент продажи товара, а не в момент оплаты поставщику. Поэтому себестоимость также определяется на основании данных о продажах и переносится в лист «Источник начисления».

Пример №3. Особенности отражения заработной платы

Отражение расходов на заработную плату требует особого внимания, так как в зависимости от ситуации одна и та же сумма может попасть в ОПиУ разными способами. Основной принцип остается неизменным: расход на заработную плату должен быть отражен в том месяце, за который она начислена, независимо от даты фактической выплаты.

Однако в работе компании возможны несколько вариантов.

Вариант 1. Стандартная ситуация

В большинстве случаев заработная плата выплачивается в том же месяце, за который она начислена, либо в первых числах следующего месяца. В этом случае дополнительное отражение в листе «Источник начисления» не требуется. Если дата выплаты относится уже к следующему месяцу, финансовый менеджер указывает дату начисления в поле «Дата начисления (ручной ввод)» банковской операции. После этого система автоматически относит расход к правильному отчетному периоду.

Пример:

Начисление Выплата Дата начисления (ручной ввод) Отражение в ОПиУ
Январь 3 февраля 31 января Январь

Вариант 2. Выплата произведена из другого источника

Иногда заработная плата выплачивается из источника, который не отражается в банковских выписках, используемых в системе. В этом случае расход все равно должен попасть в ОПиУ. Для этого начисленная сумма отражается в листе «Источник начисления». Таким образом, финансовый результат периода формируется корректно даже при отсутствии банковской операции.

Вариант 3. Банковская выписка появилась позднее

После отражения заработной платы через лист «Источник начисления» фактическая выплата может позднее появиться в банковской выписке. В этот момент возникает риск двойного отражения расхода в ОПиУ. Чтобы этого избежать, необходимо выполнить два действия.

Шаг 1. В банковской операции указать правильную дату начисления в поле «Дата начисления (ручной ввод)».

Шаг 2. В листе «Источник начисления» внести корректирующую запись с противоположным знаком (сторно), которая отменит ранее отраженное начисление.

После этого расход останется в ОПиУ только один раз.

Контроль после появления банковской операции

Если заработная плата первоначально была отражена через лист «Источник начисления», а затем появилась в банковской выписке, финансовый менеджер обязан проверить:

  • отражена ли банковская операция в системе;
  • заполнена ли «Дата начисления (ручной ввод)»;
  • внесена ли корректирующая запись в лист «Источник начисления»;
  • отсутствует ли двойное отражение заработной платы в ОПиУ.

Наиболее распространенная ошибка

Самая частая ошибка при работе с заработной платой — оставить первоначальное начисление в листе «Источник начисления» после того, как выплата уже появилась в банковской выписке. В этом случае одна и та же сумма будет учтена дважды:

  • первый раз — через лист «Источник начисления»;
  • второй раз — через банковскую операцию с датой начисления.

Результатом станет завышение расходов на оплату труда и искажение финансового результата периода.

Какие операции заносятся в лист «Источник начисления»

В настоящее время в системе F&C через лист «Источник начисления» отражаются следующие операции:

Доходы

  • Выручка от реализации товаров (на основании данных Vendus).

Расходы

  • Себестоимость реализованного товара (на основании данных Vendus);
  • Начисленная заработная плата;
  • Начисления INPS работодателя;
  • Другие операции, которые должны быть отражены по методу начисления.

По мере развития системы перечень операций может быть дополнен.

Важно понимать

Лист «Источник начисления» не является единственным источником формирования ОПиУ. Это одна из самых распространенных ошибок. ОПиУ формируется одновременно из нескольких источников.

Из каких источников формируется ОПиУ

Все операции можно разделить на пять типов.

Тип 1. Операции, полностью совпадающие с ДДС

По таким операциям:

  • совпадает сумма;
  • совпадает дата;
  • совпадает статья.

Они автоматически попадают в ОПиУ из Базы данных. Пример:

  • коммунальные услуги;
  • услуги связи;
  • аренда (если признается по дате оплаты).

В этом случае никаких дополнительных действий выполнять не требуется.

Тип 2. Совпадает сумма, но отличается дата

По таким операциям:

  • сумма одинакова;
  • статья одинакова;
  • отличается только дата признания.

В этом случае используется поле «Дата начисления (ручной ввод)». Пример: аренда оплачена в конце месяца, но относится к следующему месяцу. После указания даты начисления система автоматически перенесет расход в нужный отчетный период.

Тип 3. Есть в ОПиУ, но отсутствует в ДДС

Такие операции полностью отражаются через лист «Источник начисления». Примеры:

  • выручка;
  • себестоимость;
  • начисленная заработная плата;
  • начисление INPS работодателя.

В банковской выписке таких операций может не быть вовсе.

Тип 4. Есть в ДДС, но отсутствует в ОПиУ

Эти операции влияют только на движение денежных средств. На финансовый результат они не влияют. Например:

  • получение кредита;
  • возврат займа;
  • переводы между собственными банковскими счетами.

Для таких операций в справочнике статей устанавливается значение: Статья ОПиУ = «Не учитывать».

Тип 5. Есть и в ДДС, и в ОПиУ, но отличаются сумма и дата

Это наиболее сложная категория операций. Примером является начисление INPS работодателя. В банковской выписке отражается общая сумма налога. Однако в ОПиУ должна попасть только та часть, которая относится к расходам работодателя. В системе F&C такая корректировка отражается через лист «Источник начисления». Это позволяет одновременно сохранить корректный ДДС и правильно сформировать финансовый результат.

Когда необходимо работать с листом «Источник начисления»

Финансовый менеджер открывает этот лист после завершения обработки банковских выписок. Перед заполнением необходимо убедиться, что:

  • все банковские операции уже перенесены;
  • База данных сформирована корректно;
  • отсутствуют ошибки классификации.

Только после этого вносятся операции по методу начисления.

Что необходимо проверить после заполнения

После внесения данных необходимо убедиться, что:

  • все операции отражены в нужном периоде;
  • суммы соответствуют расчетным данным;
  • операции корректно попали в ОПиУ;
  • в ДДС не появилось дублирования.
Типичные ошибки

Ошибка №1. Внесение банковской операции в лист «Источник начисления».

Следствие: операция учитывается дважды.

Ошибка №2. Использование даты оплаты вместо даты начисления.

Следствие: искажается финансовый результат периода.

Ошибка №3. Попытка перенести через «Источник начисления» операции, которые уже автоматически отражаются по кассовому методу.

Следствие: двойной учет расходов или доходов.

Запомните: Лист «Источник начисления» не заменяет банковские листы. Он используется только для тех операций, которые невозможно корректно отразить по данным банковских выписок или которые должны быть признаны в другом отчетном периоде. Главная задача этого листа — обеспечить формирование ОПиУ по методу начисления, сохранив при этом достоверность ДДС.
Блок 7. Как система формирует управленческую отчетность ▼

После того как все хозяйственные операции внесены в рабочие листы файла «УУ_данные», система автоматически формирует управленческую отчетность. Финансовому менеджеру не требуется вручную переносить данные между листами или рассчитывать показатели. Главная задача — обеспечить полноту и корректность исходной информации. Все дальнейшие расчеты выполняются автоматически.

Последовательность формирования отчетности

Работа системы построена по определенному алгоритму. Каждый этап зависит от правильности предыдущего.

Формирование ДДС

Отчет о движении денежных средств (ДДС) показывает фактическое движение денежных средств компании. В него попадают только операции, связанные с поступлением или выбытием денежных средств. Основным источником информации являются банковские листы. В отдельных случаях могут использоваться дополнительные источники, если это предусмотрено методологией системы.

ДДС отвечает на вопросы:

  • сколько денежных средств поступило;
  • сколько денежных средств было выплачено;
  • по каким направлениям происходило движение денежных средств;
  • какой остаток денежных средств сформировался на конец периода.

Важно понимать, что ДДС не показывает прибыль компании. Он отражает только движение денег.

Формирование ОПиУ

Отчет о прибылях и убытках (ОПиУ) показывает финансовый результат деятельности компании. В отличие от ДДС, ОПиУ строится по методу начисления. Это означает, что доходы и расходы отражаются в том периоде, к которому они относятся, независимо от даты оплаты.

ОПиУ формируется одновременно из нескольких источников:

  • операций, автоматически попадающих по кассовому методу;
  • операций с использованием поля «Дата начисления (ручной ввод)»;
  • листа «Источник начисления».

Именно сочетание этих источников позволяет получить достоверный финансовый результат за отчетный период.

Почему ДДС и ОПиУ отличаются

Очень важно понимать, что различия между ДДС и ОПиУ являются нормальной частью работы системы. Совпадение показателей этих отчетов не является целью. Наоборот, в большинстве случаев суммы и даты будут различаться. Например:

  • покупатель может оплатить товар раньше его отгрузки;
  • заработная плата может быть выплачена в следующем месяце;
  • поставщик может получить оплату раньше поступления товара;
  • кредит увеличивает остаток денежных средств, но не влияет на прибыль компании.

Поэтому различия между ДДС и ОПиУ не являются ошибкой. Ошибкой считается только ситуация, когда операция отражена не в соответствии с установленной методологией.

Проверка сформированных отчетов

После обработки всех данных финансовый менеджер открывает файл «УУ_финотчеты». Для каждого отчета предусмотрено два варианта представления данных:

  • отчет по формулам;
  • сводная таблица.

Оба варианта строятся на основании одной и той же Базы данных. Поэтому итоговые показатели должны совпадать. Если итоговые суммы отличаются, это означает, что необходимо проверить настройки отчета или корректность формирования данных.

Как искать ошибки

Поиск ошибок всегда выполняется в обратном порядке формирования данных. Необходимо придерживаться следующего алгоритма:

  1. Обнаружить ошибку в отчете.
  2. Проверить соответствующую запись в Базе данных.
  3. Определить источник формирования этой записи.
  4. Исправить данные в источнике.
  5. Убедиться, что изменения автоматически отразились в Базе данных.
  6. Повторно проверить ДДС и ОПиУ.

Категорически запрещается исправлять итоговые показатели непосредственно в отчетах.

Главное правило

Отчеты являются результатом работы системы, а не местом для внесения изменений. Если финансовый менеджер видит ошибку в ДДС или ОПиУ, он должен искать причину в одном из источников данных:

  • банковских листах;
  • листе «Источник начисления»;
  • справочниках;
  • настройках классификации операций.

Только после исправления первичной информации система автоматически сформирует корректные отчеты.

Блок 8. Почему в F&C одновременно существуют два варианта отчетов ▼

В системе F&C каждый управленческий отчет представлен в двух вариантах:

  • Отчет по формулам;
  • Сводная таблица.

На первый взгляд может показаться, что оба отчета дублируют друг друга. Однако это не так. Каждый из них выполняет свою задачу.

Отчет по формулам

Отчет по формулам является основным управленческим отчетом, который используется для анализа финансового состояния компании и предоставляется собственнику. В нем данные уже сгруппированы в соответствии с утвержденной структурой отчетности. Именно этот отчет используется для анализа:

  • выручки;
  • себестоимости;
  • валовой прибыли;
  • операционных расходов;
  • чистой прибыли;
  • движения денежных средств.

Все показатели рассчитываются автоматически на основании данных Базы данных.

Сводная таблица

Сводная таблица используется не для подготовки отчетности собственнику, а для контроля правильности данных. Она позволяет быстро проверить:

  • все ли операции попали в отчет;
  • правильно ли определились статьи;
  • нет ли операций, отраженных не в том разделе;
  • корректно ли распределились суммы по месяцам;
  • отсутствуют ли дублирующиеся записи.

Кроме того, сводная таблица позволяет быстро анализировать данные по различным разрезам без изменения структуры основного отчета. По сути, сводная таблица является инструментом внутренней проверки качества управленческого учета.

Что необходимо проверять

После формирования отчетности финансовый менеджер должен убедиться, что:

  • итоговые показатели отчета по формулам соответствуют данным сводной таблицы;
  • отсутствуют необъяснимые расхождения между отчетами;
  • каждая операция отражена только один раз;
  • суммы распределены по правильным периодам;
  • данные соответствуют первичным документам.

Если обнаружено расхождение между отчетом по формулам и сводной таблицей, необходимо определить причину до передачи отчетности руководству. В системе F&C действует правило: сначала проверяется правильность данных, и только после этого анализируются финансовые показатели. Поэтому финансовый менеджер должен воспринимать сводную таблицу как рабочий инструмент контроля, а отчет по формулам — как итоговый результат своей работы. Только после того, как оба варианта отчета совпадают, отчетность считается готовой к передаче финансовому директору или собственнику компании.

Запомните: Отчет по формулам — это официальный управленческий отчет. Сводная таблица — это инструмент контроля качества данных. Они не заменяют друг друга, а работают совместно. Именно благодаря использованию двух вариантов отчетов система F&C позволяет своевременно выявлять ошибки, контролировать качество данных и обеспечивать достоверность управленческой отчетности.
